# Prism.js Generation Architecture

**Clear separation of concerns: Patterns from Parser SSOT, Colors from Theme YAML**

## Overview

Prism.js syntax highlighting for `.zolo` files requires TWO components:

1. **Syntax Patterns (JS)** - What to highlight
2. **Color Theme (CSS)** - How to color highlighted tokens

These are generated by **separate, specialized generators** with different inputs:

```
┌─────────────────────────────────────────────────────────┐
│  PATTERNS (What to highlight)                           │
├─────────────────────────────────────────────────────────┤
│  Input:  Parser SSOT (KeyDetector, TokenRegistry)       │
│  Output: 6 JS language files (zolo, zspark, zui, etc.)  │
│  Path:   zlsp/generators/generate_prism_zolo.py         │
│  Run:    python3 -m zlsp.generators.generate_prism_zolo │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  COLORS (How to color tokens)                           │
├─────────────────────────────────────────────────────────┤
│  Input:  Theme YAML (zolo_default.yaml)                 │
│  Output: prism-zolo-theme.css                           │
│  Path:   zlsp/themes/generators/prism.py                │
│  Run:    python3 -m zlsp.themes.generators.prism        │
└─────────────────────────────────────────────────────────┘
```

## Why This Architecture?

### Problem with Old Approach (DEPRECATED)

Before this refactor, `zlsp/themes/generators/prism.py` tried to do BOTH:
- ❌ Hardcoded JS patterns in Python strings
- ❌ Generated colors from theme YAML
- ❌ Treated Prism like an IDE (extended `BaseGenerator` alongside Vim/VSCode)
- ❌ No file-type specialization (zspark, zui, zschema, etc.)
- ❌ Patterns duplicated parser logic (two sources of truth)

**Result:** Architectural misalignment. Prism ≠ IDE.

### New Approach (CORRECT)

**Patterns and colors have different sources:**

| Aspect | Source | Why |
|--------|--------|-----|
| **Patterns** | Parser SSOT | Parser already knows what a "root key" or "type hint" is. Extracting from parser guarantees consistency. |
| **Colors** | Theme YAML | Color schemes are subjective and should be configurable. Theme YAML is SSOT for colors. |

**Generators aligned to their purpose:**

| Generator | Extends BaseGenerator? | Input | Output | Purpose |
|-----------|------------------------|-------|--------|---------|
| `VimGenerator` | ✅ Yes | Theme YAML | Vim highlight commands | Color mappings for IDE |
| `VSCodeGenerator` | ✅ Yes | Theme YAML | TextMate grammar + colors | Grammar + colors for IDE |
| `PrismCSSGenerator` | ✅ Yes | Theme YAML | CSS theme | **Colors only** for web |
| `generate_prism_zolo.py` | ❌ No (standalone) | Parser SSOT | 6 JS language files | **Patterns only** for web |

**Key insight:** Vim/VSCode need colors. Prism needs colors AND patterns, but from different sources.

## File Structure

```
zlsp/
├── generators/                          ← PATTERNS (from parser SSOT)
│   ├── generate_prism_zolo.py           ← Main script (6 languages)
│   ├── prism_pattern_generator.py       ← Pattern orchestration
│   ├── prism_pattern_transformer.py     ← zlsp → Prism regex
│   ├── file_type_pattern_extractor.py   ← Extract KeyDetector logic
│   └── README.md                        ← Usage guide
│
├── themes/
│   └── generators/                      ← COLORS (from theme YAML)
│       ├── prism.py                     ← CSS generation ONLY
│       ├── vim.py                       ← Vim colors
│       └── vscode.py                    ← VSCode colors + grammar
│
└── generated/                           ← Output (JS patterns)
    ├── prism-zolo.js                    ← Generic base
    ├── prism-zspark.js                  ← zSpark.*.zolo files
    ├── prism-zui.js                     ← zUI.*.zolo files
    ├── prism-zschema.js                 ← zSchema.*.zolo files
    ├── prism-zconfig.js                 ← zConfig.*.zolo files
    └── prism-zenv.js                    ← zEnv.*.zolo files
```

## Usage

### Quick Start (Recommended)

```bash
# Generate both patterns and CSS
zlsp generate-prism

# Or generate separately
zlsp generate-prism --patterns-only  # JS patterns only
zlsp generate-prism --css-only       # CSS theme only
```

## When to Regenerate

### Regenerate JS Patterns (Step 1) when:
- Parser rules change (KeyDetector, TokenRegistry)
- New token types added (TokenType enum)
- File type detection logic changes (FileTypeDetector)
- Any change to how zlsp parses `.zolo` syntax

### Regenerate CSS Theme (Step 2) when:
- Color scheme changes (zolo_default.yaml)
- New token types need color mappings
- CSS class names change

## Benefits

### 1. Single Source of Truth
- Pattern rules live ONLY in parser (KeyDetector, TokenRegistry)
- Color definitions live ONLY in theme YAML
- No duplication, no drift

### 2. Automatic Sync
- Parser changes → regenerate JS → Prism updated
- Theme changes → regenerate CSS → colors updated

### 3. File-Type Specialization
- 6 language-specific files (zspark, zui, zschema, etc.)
- Each has file-type-specific patterns from KeyDetector
- Example: `zSpark:` colors green in zspark language, pink in generic

### 4. Testability
- 94 tests validate pattern extraction/generation
- Integration tests check file synchronization
- Browser tests provide visual validation

### 5. Clear Architecture
- Prism no longer treated as an IDE
- Theme generators focus on colors (Vim, VSCode, Prism CSS)
- Pattern generators focus on syntax (SSOT-based)

## Summary

**Old architecture:** Confused Prism with IDE, hardcoded patterns, single generic language.

**New architecture:** Clear separation - patterns from parser SSOT, colors from theme YAML, 6 specialized languages.

**Result:** Maintainable, testable, aligned with zlsp philosophy.
